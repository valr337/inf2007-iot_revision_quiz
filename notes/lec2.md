# CSC2106 Lecture 2 — HTTP & REST

## 1. Where HTTP and REST fit in IoT
The lecture places HTTP in the IoT protocol stack at the **application / data layer**, alongside protocols such as MQTT and CoAP.

### Key idea
HTTP and REST are useful for IoT systems that need:
- web integration
- dashboards
- cloud APIs
- remote monitoring
- device control
- data retrieval

They are generally more suitable for **internet-connected, higher-bandwidth** IoT applications than for very constrained networks.

## 2. HTTP basics
HTTP stands for **HyperText Transfer Protocol**.

### Basic client-server flow
1. client and server establish a **TCP connection**
2. client sends an **HTTP request**
3. server returns an **HTTP response**
4. the connection is usually closed

### Important point
HTTP follows a **request-response model**.

## 3. Web content
HTTP responses return web content.

### Web content definition
A piece of web content is:
- a sequence of bytes
- with an associated **MIME type**

### MIME examples from lecture
- `text/html` → HTML document
- `text/plain` → unformatted text
- `application/postscript` → Postscript document
- `image/gif` → GIF image
- `image/jpeg` → JPEG image

## 4. Static vs dynamic content
### Static content
Stored in files and retrieved when requested.

Examples:
- HTML files
- images
- audio clips

### Dynamic content
Produced on-the-fly in response to an HTTP request.

Example:
- content generated by a program executed by the server

### Bottom line
All web content is associated with a file managed by the server.

## 5. URL, URI, and resource naming
### URL
A URL identifies a file managed by a server and can include:
- scheme
- host
- port
- path
- optional arguments

Examples shown:
- `http://www.singaporetech.edu.sg:80/index.html`
- `http://www.singaporetech.edu.sg`
- `http://www.singaporetech.edu.sg:8080/cgi-bin/simi?15432&223`

### URI vs URL
- **URI** = identifier of a resource
- **URL** = a type of identifier that also tells you how to access it

## 6. HTTP request structure
An HTTP request consists of:
- a **request line**
- zero or more **request headers**

### Request line format
`<method> <uri> <version>`

Example:
`GET /devices/123/temperature`

### Common methods from lecture
- GET
- POST
- OPTIONS
- HEAD
- PUT
- DELETE
- TRACE

## 7. HTTP methods
### GET
Used to retrieve static or dynamic content.

Notes:
- arguments for dynamic content are often in the URI
- described as the main “workhorse” method

### POST
Used to retrieve or submit dynamic content with arguments in the request body.

In REST usage, POST is commonly used to **create** a resource.

### OPTIONS
Gets server or file attributes.

### HEAD
Like GET, but **no response body**.

### PUT
Writes a file to the server.
In REST usage, PUT is used to **replace / update an entire resource**.

### DELETE
Deletes a file / resource from the server.

### TRACE
Echoes the request in the response body.
Useful for debugging.

## 8. HTTP response structure
An HTTP response consists of:
- a **response line**
- zero or more **response headers**
- optionally a **response body**

### Response line format
`<version> <status code> <status msg>`

### Status codes shown in lecture
- `200 OK` → request handled successfully
- `403 Forbidden` → server lacks permission
- `404 Not Found` → file not found

### Important response headers
- `Content-Type` → MIME type of response body
- `Content-Length` → body length

## 9. IoT HTTP transaction example
The lecture gives an example of an IoT device using telnet to send:

`POST /devices/123/temperature HTTP/1.1`

with headers such as:
- `Host`
- `Content-Type: application/json`
- `Content-Length`

and a JSON payload such as:
```json
{ "device_id": "123", "temperature": "26.5" }
```

Example server response:
```http
HTTP/1.1 201 Created
Content-Type: application/json
Connection: close
```

This shows a simple HTTP-based cloud ingestion pattern for IoT sensor data.

## 10. REST basics
REST stands for **Representational State Transfer**.

### Lecture points
- introduced by **Roy Fielding** in 2000
- based on web standards
- commonly uses **HTTP**
- **not a protocol**
- it is an **architectural style**

### Key REST idea
Everything important is exposed as a **resource** and accessed through HTTP.

## 11. REST and HTTP
REST builds on the success of the Web by using:
- URI-addressable resources
- HTTP request-response interaction
- more than just GET and POST
- also PUT and DELETE when needed

## 12. Six architectural constraints of REST
1. **Client-Server Architecture**  
   Client and server act independently.

2. **Statelessness**  
   The server does not store client session state between requests.

3. **Cacheable**  
   Responses can indicate whether data may be cached.

4. **Layered System**  
   The application should behave consistently even with intermediaries.

5. **Uniform Interface**  
   Interaction is standardized and predictable.

6. **Code on Demand (optional)**  
   Clients may extend functionality by downloading executable code.

## 13. Resources in REST
A resource is the key abstraction in REST.

Examples of resources:
- a document
- an image
- today's weather in Berlin
- a collection of resources
- a person
- an IoT device

### Resource identification
Resources are represented using global identifiers, typically **URIs** in HTTP.

## 14. Naming resources
REST uses hierarchical resource naming.

Examples from lecture:
- `/books/` → retrieve all books
- `/books/ISBN-0011` → retrieve a specific book
- `/books/ISBN-0011/authors` → retrieve authors for that specific book
- `/classes/cs2650/students`

### Principle
As the path becomes more specific, you are navigating deeper into the data.

## 15. REST verbs and CRUD mapping
### GET
- retrieve items from a resource
- **safe**
- **idempotent**
- usually cacheable

### POST
- create a new item in a resource
- **not idempotent**

### PUT
- replace an existing item / resource
- **idempotent**

### PATCH
- update part of an existing item
- partial update

### DELETE
- delete an existing item
- **idempotent**

## 16. Safe vs idempotent
### Safe
A method does not change the resource state.
- example: GET

### Idempotent
Making the same request multiple times leads to the same final state.
- examples: GET, PUT, DELETE
- POST is generally **not** idempotent

## 17. REST examples from lecture
### GET example
`GET /books/ISBN-0011021`
- retrieves a specific book
- response example: `200 OK`

### POST example
`POST /books`
- creates a new book
- response example: `201 Created`

### PUT example
`PUT /books/isbn-111`
- replaces the entire existing resource
- response example: `200 OK`

### PATCH example
`PATCH /books/isbn-111`
- updates part of the resource, such as the title only
- response example: `200 OK`

### DELETE example
`DELETE /books/isbn-111`
- deletes the resource
- response example shown: `204 No Content`

## 18. Representations
REST returns **representations** of resources.

### Typical formats
- JSON
- XML

Important note from lecture:
- JSON / XML are **not the resource itself**
- they are only ways to **represent** the resource

## 19. RESTful API best practices for IoT
The lecture recommends:
- use clear hierarchical resource names  
  example: `/devices/{id}/status`
- keep APIs stateless
- prefer lightweight data formats such as JSON
- secure endpoints using authentication and HTTPS

### Security note
Exposing device control over HTTP is risky unless properly secured.

Recommended protections:
- authentication
- HTTPS
- access control
- input validation

## 20. Limitations of HTTP for IoT
### 20.1 High overhead
HTTP uses relatively large, text-based headers, increasing message size and complexity.

### 20.2 Request-response model
It is synchronous and less suitable for:
- real-time systems
- event-driven communication

It may force devices to poll, increasing traffic and energy usage.

### 20.3 Latency and connection management
Each request may require a new TCP connection, increasing:
- latency
- power usage

### 20.4 Not optimized for constrained networks
Protocols like LoRa and Zigbee do not directly support HTTP due to its overhead.

### 20.5 Security cost
Plain HTTP sends data in plaintext.
HTTPS improves security but adds computational and bandwidth cost, which can be hard for constrained devices.

## 21. HTTP vs MQTT vs CoAP
### HTTP
- model: request-response
- overhead: high
- bandwidth usage: high
- suitability: web apps, dashboards
- power usage: high
- security: HTTPS / TLS
- reliability: reliable due to TCP
- scalability: moderate

### MQTT
- model: publish-subscribe
- overhead: low
- suitability: sensors, telemetry, alerts
- power usage: low
- reliability: QoS 0–2
- scalability: high

### CoAP
- model: request-response
- overhead: very low
- suitability: constrained devices, M2M
- power usage: very low
- security: DTLS
- scalability: high

## 22. Final summary
### Definitions
- HTTP = web communication protocol using request-response
- REST = architectural style leveraging HTTP for structured APIs

### IoT relevance
RESTful APIs are useful for:
- remote monitoring
- device control
- data retrieval
- dashboard / cloud integration

### Main challenge
HTTP works well for high-bandwidth, always-on networks, but is less suitable for:
- low-power devices
- lossy networks
- highly constrained environments

### Alternatives
- **MQTT** for lightweight publish-subscribe communication
- **CoAP** for constrained, REST-like device communication
